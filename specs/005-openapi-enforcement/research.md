# Phase 0 Research — OpenAPI auto-generation & enforcement

## R1. How to derive the API surface without annotations

**Decision**: Introspect the live Flask app built by `koan/app/api/__init__.py::create_app()`
using `app.url_map.iter_rules()` and `app.view_functions`.

- Each Werkzeug `Rule` yields `.rule` (e.g. `/v1/missions/<mission_id>`), `.methods`
  (a set incl. auto-added `HEAD`/`OPTIONS`), and `.endpoint` (e.g. `missions.get_mission_route`).
- Convert Werkzeug path syntax to OpenAPI: `<mission_id>` → `{mission_id}`,
  `<converter:name>` → `{name}` (drop the converter). Regex:
  `re.sub(r"<(?:[^:<>]+:)?([^<>]+)>", r"{\1}", rule)`.
- Filter out `HEAD` and `OPTIONS` (Flask-injected, not part of the documented contract).
- The endpoint's blueprint (the part before the first `.`) becomes the OpenAPI `tag`.

**Rationale**: The route table is the single source of truth for *what exists*. A generator
that reads it cannot describe a route that isn't registered, and automatically reflects
additions/removals (FR-002, FR-004). No decorators to add across 22 handlers, no framework
migration (Principle VII).

**Alternatives rejected**:
- `flask-smorest` / `apispec` + marshmallow schemas — would require annotating every handler
  and adding dependencies; large surface, high drift risk between schema and handler, contrary
  to the low-dependency posture. Deferred indefinitely.
- Hand-authored `openapi.yaml` — exactly the drift problem this feature exists to kill (FR-002).

## R2. How to know which routes require auth

**Decision**: Make `require_token` (in `koan/app/api/auth.py`) tag its wrapper function with
`decorated._koan_requires_token = True`. The generator reads
`app.view_functions[endpoint]` and treats the operation as secured iff that attribute is truthy.

**Rationale**: Auth-required status then has exactly one source of truth — the decorator that
actually enforces it (Principle VI). A new secured route gets `security: [{bearerAuth: []}]`
for free; a new public route (like `/v1/health`, which is undecorated) gets `security: []`.
No hand-maintained allow-list of public/secured endpoints that could silently rot.

**Alternatives rejected**:
- Hardcode "everything except `/v1/health` is secured" — brittle; a future public endpoint
  would be mislabeled as secured, and a mistakenly-undecorated secured endpoint would still be
  documented as secured, hiding the bug.
- Inspect `__wrapped__` closures — fragile and implementation-coupled.

## R3. Serialization & determinism

**Decision**: Serialize with PyYAML (`yaml.safe_dump`) using
`sort_keys=True, default_flow_style=False, allow_unicode=True, width=100`. Prepend a fixed
`# Generated by `make openapi` — do not edit by hand.` header. Sort methods and any list we
control. Pin `info.version = "1.0.0"` (the `/v1` contract version), independent of
`app.__version__`.

**Rationale**: Byte-identical output on unchanged code is the linchpin of enforcement
(FR-006) — any nondeterminism makes the drift check flap. `sort_keys=True` removes dict
iteration-order sensitivity; pinning the version removes release-bump churn.

**Alternatives rejected**:
- JSON — valid, but YAML diffs are more reviewable and the API doc audience reads YAML.
  A JSON sibling can be added later (deferred).
- `ruamel.yaml` round-trip — unnecessary; we generate fresh each time, no comments to preserve.

## R4. Where the artifact lives & how CI scopes the check

**Decision**: Commit the artifact at `koan/openapi.yaml`. CI workflow
`.github/workflows/openapi.yml` triggers on `pull_request` with
`paths: [koan/app/api/**, koan/openapi.yaml, Makefile, .github/workflows/openapi.yml]`.

**Rationale**: `koan/app/api/**` covers all current + future route modules and the generator
itself with one glob (FR-009). Adding the artifact + Makefile + the workflow file to the
filter means edits that could change generation are caught, while unrelated PRs skip the job
entirely (FR-008) — GitHub does not launch a workflow whose `paths` filter matches nothing.

**Alternatives rejected**:
- Enumerate each route file — misses newly added modules (explicitly forbidden by FR-009).
- Run on every PR — wastes CI minutes; violates the "only when API updated" requirement.
- `paths-ignore` — inverts the logic and is easy to get wrong; `paths` (allow-list of
  API-defining globs) states intent directly.

## R5. Generation must not need a running server or a token

**Decision**: The generator constructs the app via `create_app(koan_root=<temp>)` inside the
process, reads the route table, and never calls `app.run()` / binds a port / reads a token.

**Rationale**: FR-003/FR-005 — generation is a static introspection. `create_app()` only
needs a `koan_root` to exist (it `mkdir`s a `logs/` dir); the generator passes a throwaway
temp dir so it never pollutes the working tree and never depends on `instance/` or config.

## R6. Validity check

**Decision**: Tests assert the document parses as YAML, has top-level `openapi` (starts with
`3.`), `info.version`, non-empty `paths`, a `components.securitySchemes.bearerAuth` of type
`http`/scheme `bearer`, and a `components.schemas.Error`. A full JSON-Schema validation via an
external validator is out of scope (no new dep); structural assertions cover FR-012 adequately.

**Rationale**: Keeps the dependency footprint flat while still guarding the document's shape.
